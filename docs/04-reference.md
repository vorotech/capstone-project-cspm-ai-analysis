# Довідник: Категорії та Утиліти

## Легенда категорій (Adjustment Categories)
Під час аналізу LLM використовує жорстко задану таксономію для пояснення причини зміни критичності або позначення знахідки як False Positive:
- **`COMPENSATING_CONTROL`**: Знайдено інший ресурс (наприклад, WAF, Security Group, KMS), який нівелює або зменшує ризик, на який скаржиться CSPM.
- **`ISOLATED_ENVIRONMENT`**: Ресурс знаходиться у приватній підмережі або повністю ізольований від інтернету, що значно знижує вектор атаки.
- **`BUSINESS_REQUIREMENT`**: Відкрита конфігурація є очікуваною бізнес-логікою (наприклад, публічний S3 бакет для статичного веб-сайту).
- **`TOOL_INACCURACY`**: Класичний False Positive, коли CSPM інструмент помилився або не підтримує глибокий аналіз конкретного типу ресурсів.
- **`TEST_RESOURCE`**: Архітектура вказує на те, що це тестове або некритичне середовище, для якого суворі правила є надлишковими.
- **`ACCEPTED_RISK`**: Ризик мінімальний з огляду на загальний контекст і може бути класифікований як прийнятний (Accepted Risk).
- **`THIRD_PARTY_MANAGED`**: Ресурс або конфігурація керується зовнішнім SaaS/PaaS рішенням, яке не підлягає прямому контролю або модифікації інженером.
- **`INCREASED_CONTEXTUAL_RISK`**: Узагальнена категорія для підвищення критичності, коли загальний контекст робить проблему небезпечнішою.
- **`CRITICAL_BUSINESS_FUNCTION`**: Підвищення критичності через те, що ресурс виконує життєво важливу бізнес-функцію і його компрометація призведе до зупинки системи.
- **`PUBLICLY_EXPOSED_ASSET`**: Підвищення критичності через те, що ресурс безпосередньо відкритий в інтернет, хоча базова оцінка CSPM цього не враховує.
- **`SENSITIVE_DATA_EXPOSURE`**: Підвищення критичності через те, що ресурс обробляє або зберігає конфіденційні дані (PII, фінансова інформація тощо).
- **`NONE`**: Жодних змін критичності не було застосовано.

## Оцінювані Метрики
Під час дослідження для кожної LLM обчислюються наступні ключові показники:
- **`Adjustment Rate (%)`**: Відсоток знахідок CSPM, для яких модель змінила критичність або позначила їх як False Positive. Показує, наскільки часто модель "не погоджується" з базовою оцінкою сканера.
- **`Decision Consistency (%)`**: Стабільність рішень моделі. Показує, наскільки часто модель для однієї і тієї ж знахідки (в різних запусках) повертає ідентичну оцінку: ту саму фінальну критичність, статус False Positive та причину виправлення.
- **`OK Status (%)`**: Відсоток успішних запусків (відсутність таймаутів, помилок парсингу JSON з відповіді LLM, тощо).


## Корисні утиліти (Utilities)
У проєкті передбачено кілька скриптів для керування даними та історією запусків:

- **Повне очищення історії (Старт з нуля):** Якщо ви хочете повністю видалити всі зібрані дані, історію LLM-аналізів та проміжні файли CSPM, просто видаліть папку `output/` вручну (наприклад, `rm -rf output/`). Пайплайн автоматично створить її заново при наступному запуску.
- `python scripts/remove_model_history.py --model <MODEL_NAME>`: **Видалення конкретної моделі.** Дозволяє вилучити результати (а отже і вплив на загальну статистику та графіки) конкретної моделі, якщо вона генерує сміття чи галюцинує.
- `python scripts/fetch_nist_mapping.py`: **Завантаження офіційного NIST OSCAL.** Скрипт завантажує базу даних контролів `800-53-rev5` та формує локальний мапінг `data/nist_800_53_rev5_mapping.json`. Використовується для збагачення знахідок без використання LLM.

## Аналітичні скрипти (Тека `analysis/`)
У проєкті є два ключові файли для обробки зібраних даних, що виконують різні функції:
- `analysis/results_analysis.ipynb`: Дослідницьке середовище (Exploratory Data Analysis). Генерує складні графіки, таблиці та текстові звіти. Результат експортується в HTML для відображення як "Додаток" у фінальній презентації.
- `analysis/generate_summary.py`: Скрипт-міст (генератор API) між даними та React-презентацією. Він бере сирі CSV, підраховує усереднені фінальні метрики (Adjustment Rate, Decision Consistency) та зберігає їх у `presentation/public/summary.json`. Саме цей файл дозволяє фронтенду малювати інтерактивні графіки Recharts без прямого доступу до сирих даних.
